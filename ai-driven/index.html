Phase 0 — AI×BDD Discovery

先把「要做什麼」講清楚。這裡 AI Agent 是主力，人只做把關。

1PRD Input

PM、BE、Designer 對齊需求，產出原始 PRD——所有後續產出物的單一事實來源。

PMBEDesigner
2BDD 詞入格組寫

AI Agent 把需求拆成 BDD 詞入格，把模糊的描述結構化成「角色／行為／預期」。

AI AgentRD
3Gherkin Generation

由詞入格生成 Gherkin 場景（Given / When / Then），成為人與機器都讀得懂的規格。

AI AgentPMQA
4Consistency Check

檢查場景有沒有漏、有沒有互相矛盾。場景不足就回流重寫，品質閘設計進流程裡。

AI AgentRD

Phase 1 — OpenNet TDD

把場景翻成測試藍圖。這是動手寫程式前的最後一道閘。

512-Ch TDD Design

Architect 主導，把場景轉成一份 12 章節的測試設計，定義測試結構與覆蓋範圍。

AI AgentArchitectRD
6Kickoff Review

PM、RD、BE、QA 全員對齊才開工。對不上就退回設計，避免做到一半才發現認知有落差。

PMRDBEQA

Phase 2 — Implementation

先寫測試、再寫實作。RD 與 AI Agent 配對，測試全綠才算完成。

7TDD Red 寫測試

先寫會失敗的測試，確認測試真的在測該測的東西——紅燈是預期、是起點。

RDAI Agent
8TDD Green 實作

寫實作到測試全過（綠燈）。沒過就回流——程式碼永遠由測試定義「做完了沒」。

RDAI Agent

兩個設計亮點

面試 / demo 時一定要點到的兩件事。

🔴 紅虛線回流＝把品質閘設計進流程

當 Consistency Check 發現場景不夠完整，不會硬往下做，而是退回重寫場景。 品質不是靠人記得，而是流程本身擋下來的——每一道閘不過就回流上一步。

🟡 角色標籤＝AI 與人的分工

每個節點下方的標籤是參與角色、依工作量排序。你會看到 AI Agent 集中在前段 （Discovery／設計），人在後段把關。這就是「AI×人」分工的具體落點。

這套方法的核心價值

整套的價值在於：需求 → 場景 → 測試 → 實作有一條可追溯的線 （AC ↔ Gherkin ↔ Test 雙向對應），任何一步品質不過就回流。

所以交付出來的東西——規格、測試、程式碼永遠是一致的， 而且事後能回溯每個決定當初的脈絡。這是 OpenNet 與 AI×BDD 兩套方法整合後的成熟版本。
